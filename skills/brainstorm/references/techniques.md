# Divergence & Grounding Toolkit

Loaded when directions feel samey, the frame feels stuck, or feasibility
needs proof. Pick the 2–3 techniques that fit — never run the list.

## Reframing

**How might we…** — rewrite the ask until it names the need.
"Build a dashboard" → "How might operators know what changed since
yesterday without looking?"

**First principles** — list every assumption, delete each in turn, see what
becomes possible. "Notes need folders" → deleted → organization emerges
from content.

**Start from the experience** — write the moment of use first (the
screenshot, the CLI transcript, the five-second story), then work backwards
to the technology. Never the reverse.

## Divergence

**Inversion** — design the worst version deliberately; each failure,
reversed, is a direction. "How would we guarantee nobody trusts this
data?" → hide freshness → direction: freshness as the hero metric.

**Analogy transfer** — how does the best-in-class of a distant domain solve
this? "Onboarding that seats you like a great restaurant" beats
"onboarding like other apps".

**Constraint play** — remove the binding constraint (no budget, no legacy),
then make it 10x harsher (offline, one screen, zero config). Both ends
produce directions the comfortable middle never finds.

**Eight fast concepts** — force eight one-line concepts in a single pass
before judging any. The first three are everyone's; the last three are
yours.

**10x, not 10%** — describe the version an order of magnitude better,
simpler, or faster; then find the smallest slice of it that ships.

## Taste filters — converge

**Say no** — deciding what not to do is the craft. List seven tempting
features the direction excludes; if cutting none of them hurts, the
direction isn't sharp yet.

**Uniqueness test** — if the direction's one-liner would fit a
competitor's homepage, keep diverging.

**Perfection bar** — could a craftsman be proud of the seams — empty
states, error paths, latency — not just the demo path?

## Grounding

**Riskiest assumption first** — name the assumption that kills the
direction if false; test that one before polishing anything else.

**Docs sweep** — read the platform's current official docs and release
notes before judging feasibility: capabilities ship monthly, memory is
stale, and the strongest directions often ride something newly possible.

**Timeboxed spike** — when docs can't answer: a throwaway probe, one
question, hard timebox, result recorded in ADR.md. The code is deleted;
the answer is kept.
